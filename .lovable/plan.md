
## OpenDraft — Business Plan & Product Vision (Revised March 2026)

**The first app store where AI agents autonomously discover, bid on, and purchase software.**

---

### Vision

OpenDraft is building the infrastructure layer for AI-to-AI commerce. Today's AI agents — from Claude to GPT to custom enterprise swarms — need tools, templates, and micro-apps to accomplish tasks. There is no marketplace where they can programmatically browse, evaluate, negotiate, and purchase software. OpenDraft fills that gap.

Our long-term vision: **become the default procurement layer for the autonomous economy** — the "GitHub for vibe coders" and the "App Store for AI agents," simultaneously.

## Self-Healing & Self-Improving Apps — Implementation Plan

### Architecture

#### New Database Tables
1. **`project_goals`** — stores the "initial prompt" / intent per listing per user
   - `id`, `user_id`, `listing_id`, `goals_prompt` (text), `structured_goals` (jsonb), `created_at`, `updated_at`
2. **`improvement_cycles`** — logs each autonomous improvement attempt
   - `id`, `listing_id`, `user_id`, `trigger` (text: fork_request, cron, manual, deploy_check), `screenshot_url`, `analysis` (jsonb), `suggestions` (jsonb), `status` (pending, approved, applied, rejected), `created_at`
3. **`improvement_changes`** — individual code changes per cycle
   - `id`, `cycle_id` (FK → improvement_cycles), `file_path`, `change_type`, `description`, `code`, `risk_level`, `approved` (boolean), `applied_at`

### Phase 1: Fork Auto-Build (MVP) ✦
**Trigger:** Fork request submitted → AI auto-generates the forked app

1. **Goals Capture** — Fork request description becomes the goals prompt; stored in `project_goals`
2. **Auto-Build Edge Function** (`swarm-fork-autobuilder`)
   - Receives fork request ID
   - Loads parent listing source (ZIP from storage or GitHub)
   - Sends source + fork description + parent goals to Gemini 2.5 Pro
   - AI generates modified source code with requested features
   - Packages as new ZIP, uploads to storage
   - Creates a new draft listing linked to the fork request
   - Notifies builder with suggested changes for review
3. **Builder Review UI** — Dashboard tab showing AI-generated fork with diff view
   - Approve all / approve individual changes / reject
   - On approve → listing goes live, fork request marked "delivered"

### Phase 2: Screenshot Analysis Loop
**Trigger:** Daily cron + on deploy + manual "Improve" button

1. **Screenshot Capture** — Reuse existing Firecrawl/Lambda screenshot pipeline
2. **AI Analysis Edge Function** (`swarm-app-analyzer`)
   - Takes screenshot + goals prompt
   - Compares current state vs intended goals using Gemini 2.5 Flash (vision)
   - Generates structured improvement suggestions
3. **Improvement Dashboard** — Shows screenshot + AI suggestions side-by-side
   - Each suggestion has risk level, code diff preview
   - User approves/rejects individual suggestions

### Phase 3: Auto-Apply & Self-Healing
**Trigger:** On deploy health check failures

1. **Health Monitor Enhancement** — Extend site-doctor to check against goals
2. **Auto-Fix Pipeline** — Low-risk fixes auto-apply (CSS, copy, layout)
3. **Continuous Improvement** — Track improvement history, learn from rejections

### Key Design Decisions
- **Autonomy:** Suggest & approve — user always has final say
- **Goals storage:** Separate `project_goals` table (editable, versioned)
- **AI models:** Gemini 2.5 Pro for code generation, Gemini 2.5 Flash for screenshot analysis
- **Screenshot pipeline:** Reuse existing Firecrawl + Lambda infrastructure
- **Triggers:** All four — fork request, daily cron, manual button, deploy health check
